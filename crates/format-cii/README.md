# invoicekit-format-cii

UN/CEFACT Cross Industry Invoice D16B reader and writer for the InvoiceKit invoice model, with a lossiness ledger on every parse.

## What it does

This crate maps the CII D16B `CrossIndustryInvoice` syntax — the XML carried by Factur-X and ZUGFeRD — to and from `invoicekit_ir::CommercialDocument`. Reading a document gives you the parsed model plus a `LossinessLedger` that records exactly what happened to every field. Writing produces deterministic, canonicalized XML: the same document serializes to identical bytes on every run, because the output is passed through `invoicekit_canonical::canonicalize_xml` before it is returned.

CII has many standard fields that the core invoice model does not yet have a home for. This crate does not drop them and does not overload them onto unrelated InvoiceKit fields. Instead it preserves them as canonical raw XML fragments inside a CII document-fields extension, so a parse/serialize cycle keeps the data byte-for-byte. The deliberate decisions about which fields are preserved this way — and why each one is not, say, an InvoiceKit tenant or trace identifier — are written down as named constants in the `mapping` module rather than left implicit.

The supported document types are invoice (CII type code 380) and credit note (381). Anything else returns a typed error rather than a guess.

## Where it sits

In the pipeline `engine -> ir -> canonical -> format/profile -> validate -> render/intake -> transmit -> evidence`, this is a format crate. It sits between the invoice model and the wire: it reads CII XML up into `CommercialDocument`, and writes `CommercialDocument` back down to CII XML. It depends on `invoicekit-ir` for the model and `invoicekit-canonical` for the byte-stable output, and on `quick-xml` for parsing. It is the read/write half for the CII syntax family; profile-specific validation (Factur-X, XRechnung, EN 16931) lives in the validate and profile layers, not here.

## Public API

Functions:

- `from_xml(input: &str) -> Result<(CommercialDocument, LossinessLedger), CiiError>` — parse a CII D16B document. Returns the model and a ledger of preserved, lost, and warned fields. The ledger is built by serializing the parse result back to XML, reparsing it, and comparing, so it reflects an actual round trip rather than a claim.
- `to_xml(document: &CommercialDocument) -> Result<String, CiiError>` — validate the document and serialize it to deterministic, canonicalized CII XML.
- `crate_name() -> &'static str` — returns `"invoicekit-format-cii"`.

Errors:

- `CiiError` — a typed enum covering malformed XML, a non-`CrossIndustryInvoice` root, wrong CII namespace for an element, unsupported type codes and document types, missing required elements, unparseable decimals and dates, IR validation failures, canonicalization failures, and invalid preserved-fragment payloads. Each variant carries a hint pointing at the likely fix.

The `mapping` module documents the CII coverage and the preservation decisions:

- `CiiMappingDecision` — one named decision: the CII element, its current representation class, the concrete IR path or extension field used, and the rationale.
- `NAMED_MAPPING_DECISIONS` — the list of those decisions (for example, `BuyerReference` is kept as a document-field extension, never as a tenant identifier; CII test indicators are profile context, not runtime flags).
- Constants pinning the source schema and extension identifiers: `CII_D16B_SCHEMA_REPOSITORY`, `CII_D16B_SCHEMA_TAG`, `CII_D16B_SCHEMA_COMMIT`, `CII_DOCUMENT_FIELDS_EXTENSION_URN`, `CII_PROFILE_CONTEXT_EXTENSION_URN`, `INVOICEKIT_CII_METADATA_EXTENSION_URN`.

The full element-by-element coverage matrix is the generated artifact at `data/cii-d16b-element-coverage.json`, derived from the pinned CEN EN 16931 schema bundle.

## Usage

```rust
use invoicekit_format_cii::{from_xml, to_xml};

let (document, ledger) = from_xml(cii_xml)?;

// Nothing was dropped or downgraded on the way in.
assert!(ledger.lost.is_empty());
assert!(ledger.warnings.is_empty());

// Write it back out. Deterministic and canonical: serializing twice
// returns identical bytes.
let xml = to_xml(&document)?;
assert!(xml.contains("<rsm:CrossIndustryInvoice"));
# Ok::<(), invoicekit_format_cii::CiiError>(())
```

## Test corpus

The crate is exercised against a committed synthetic CII D16B corpus (50 fixtures) under `conformance-corpus/synthetic/cii-d16b-profiled/`. The regression suite asserts that every fixture round-trips with a zero-loss ledger, that the serializer is deterministic and its output is already canonical, and that parse -> serialize -> parse produces an identical model. The corpus is generated by `examples/generate_cii_corpus.rs`.

## Scope

This is a syntax reader/writer for CII D16B invoices and credit notes against the current core invoice model. It is not a profile validator, and the IR surface it maps is the core model as it exists today — CII fields beyond that surface are preserved rather than interpreted. As the invoice model grows, fields move from "preserved as raw XML" to "mapped into the model", and the `mapping` decisions are where that boundary is recorded.
