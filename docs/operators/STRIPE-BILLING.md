# Stripe billing for InvoiceKit's own customers (T-140)

InvoiceKit's hosted convenience layers (managed Engine ABI,
sending, archive, certificates) are billed via Stripe. This file
is the engineering and operator runbook for wiring that up. The
core Rust + WebAssembly toolkit stays free under Apache 2.0; this
runbook only covers the hosted offering.

## Subscription plans (T-140 strict gate)

| Plan | Price | Documents/month | Hosted features |
| --- | ---: | ---: | --- |
| Trust Toolkit Free | €0 | unlimited (self-hosted) | None — runs against your own infra. |
| Hosted Starter | €49/mo | 500 sent / 500 inbound | Managed Engine ABI; sandbox AS4. |
| Hosted Team | €199/mo | 5 000 sent / 5 000 inbound | Adds managed Peppol AP, evidence-bundle archive, 30-day retention. |
| Hosted Scale | €799/mo | 50 000 sent / 50 000 inbound | Adds 7-year WORM archive, customer-managed signing key, named CSM. |
| Enterprise | custom | custom | Single-tenant pod, ISO 27001 letter, custom DPA. |

All plans bill monthly in arrears (Stripe metered subscriptions
with a flat base + per-document overage). Annual plans get a 15%
discount via a Stripe coupon code generated by the sales workflow.

Plan slugs (Stripe product IDs) live in
`services/billing/plans.toml` (follow-up bead — not in this PR);
they are referenced by the webhook handler below.

## Webhook handler

`services/billing-webhook` (follow-up scaffold) is a small Axum
service that receives Stripe webhooks at `/stripe/webhook` and
maps each event to an InvoiceKit billing-state transition. The
events we care about:

| Stripe event | InvoiceKit reaction |
| --- | --- |
| `customer.subscription.created` | Provision the customer's hosted-tier resources (Engine ABI rate limits, archive retention). |
| `customer.subscription.updated` | Adjust the tier mid-cycle; emit a notification to the customer's primary contact. |
| `customer.subscription.deleted` | Schedule the customer's tier for downgrade at period end. |
| `invoice.payment_succeeded` | Mark the period as paid; clear any "billing-blocked" flag on the tenant. |
| `invoice.payment_failed` | Set the "billing-blocked" flag on the tenant; the Engine ABI continues to serve but starts queueing transmissions. |
| `invoice.upcoming` | Surface an in-app notification on the customer dashboard; T-135 owns the dashboard. |

Webhook signature verification uses Stripe's
`Stripe-Signature` header with HMAC-SHA256 over the raw request
body. Failure to verify returns HTTP 400; we never mutate state
on an unverified webhook.

## Customer self-service portal

Stripe's hosted Billing Portal covers the customer-side flows we
don't want to rebuild:

- Update payment method.
- Cancel or downgrade subscription.
- Download paid invoices (Stripe's own invoices, not the customer's
  documents — those live in the InvoiceKit archive).
- Update billing email.

The portal session is launched from the InvoiceKit customer
dashboard (T-135) via a `POST /billing/portal-session` endpoint
that returns the Stripe portal URL. The dashboard redirects the
user to that URL; Stripe redirects back to
`https://app.invoicekit.dev/account/billing` once the user is done.

## One-time operator setup

1. **Create the Stripe account.** Use the company legal entity; the
   shipping address determines tax registration eligibility.
2. **Configure Stripe Tax** so VAT / GST is auto-calculated on
   subscription invoices. We do not handle tax ourselves
   (AGENTS.md explicit decision — no tax engine).
3. **Define the products + plans** in the Stripe Dashboard
   matching the table above. Copy the Stripe price IDs into
   `services/billing/plans.toml`.
4. **Generate restricted API keys**:
   - `STRIPE_API_KEY_SERVER` — server-side secret key with
     access to subscriptions, invoices, customers, billing
     portal sessions.
   - `STRIPE_WEBHOOK_SECRET` — Stripe webhook signing secret for
     the endpoint configured in step 5.
5. **Register the webhook endpoint** in the Stripe Dashboard with
   the URL `https://api.invoicekit.dev/stripe/webhook` and select
   the events listed in the table above.
6. **Provision a Sigstore-style OIDC** for the billing-webhook
   service so it can publish its own cosign-signed events to the
   evidence bundle — this is how the trust-toolkit
   demonstrates the billing event chain is auditable.

## Why we don't ship the actual billing service today

The trust-toolkit pitch leans on the *free* core. Adding a
billing service before there are paying customers would invert
the priority — the hosted features T-135 (customer dashboard),
T-091 (partner AP), T-080 (evidence bundle archive) are the
prerequisites. T-140 sits behind them in the build order; this
runbook captures the contract so the actual service can ship in a
single focused PR once the prerequisites are in.
